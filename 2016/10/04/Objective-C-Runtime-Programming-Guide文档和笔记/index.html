介绍

Objective-C（下简称 OC） 把尽可能多的决策从编译、链接时期推迟到运行时期。它希望尽可能的动态处理事物。这意味着 OC 不仅需要编译器，还需要一个执行编译代码的运行时系统（runtime system）。Runtime system 对于 OC 而言就像一个操作系统，让它能正常工作。

本文着眼于 NSObjcet 类，介绍 OC 程序如何与 runtime system 交互，尤其是 runtime system 如何在运行时动态的加载新类，并将消息转发给其他对象。本文还介绍了如何在程序运行时找到对象的信息的方法。

Runtime 的版本和平台

OC 的 runtime 在不同平台上有不同的版本。

历史版本和最新版本

OC 的 runtime 有两个版本，“modern”版是随着 OC 2.0 的出现而推出的，包含了很多新特性。 “legacy”版的 runtime 在 OC 1 里面有介绍。“modern”版的 API 可以查阅 Objective-C Runtime Reference。

在 modern runtime 里面最值得注意的特性就是实例变量是“稳固的”：

  • 在 legacy 版的 runtime 中，如果你改变类中实例变量的布局，你需要重新编译一下这个类的子类

  • 在 modern 版的 runtime 中，如果你改变类中实例变量的布局，你不在需要重新编译这个类的子类

另外，modern 版的 runtime 支持为声明的属性进行实例变量的合成。

平台

iPhone 应用程序和在 OS X v10.5 及以上版本上的64位程序使用 modern runtime。
其他程序（OS X 桌面上的32位程序）使用 legacy runtime。

与 Runtime 的交互

OC 程序和 Runtime 有三种不同的层次的交互：通过 OC 源代码；通过 Foundation 框架中的 NSObject 类中定义的方法；通过直接调用 runtime 方法。

Objective-C 源代码

大多数情况下 runtime system 都是在幕后自动工作的，你只需要通过编写和编译 OC 源代码就可以使用它。

当你编译你的 OC 代码时，编译器会创建能实现语言动态特性的数据结构和函数调用。数据结构捕获类（class）和分类（category）定义以及协议（protocol）声明中的信息，其中包括《The Objective-C Programming Language》中 Defining a Class and Protocols 章节讨论的类和协议，也包括方法选择器、实例变量模板和从源码中提取的其他信息。runtime 函数的主要功能是发送消息（Messaging），它由源代码的消息表达式调用。

NSObject 方法

Cocoa 中大部分的对象都是继承自 NSObject 类，所以大部分的对象都继承了 NSObject 类中所定义的方法（NSProxy 类是个例外，更多信息参见消息转发）。因此，NSObject 类的方法建立了每个实例和每个类对象固有的行为。然而，在少数情况下，NSObject 类只定义了一个模版，并不提供具体实现代码。

比如说 NSObject 类中有一个 description 的实例方法，用来返回一个字符串描述这个类。这主要用于调试 —— GDB 打印对象（print-object）命令打印从此方法返回的字符串。 NSObject 的这个方法的实现不知道类包含什么，所以它返回一个带有对象的名称和地址的字符串。 NSObject 的子类可以实现此方法以返回更多详细信息。 例如，Foundation 类 NSArray 返回它包含的对象的描述列表。

一些 NSObject 方法只是查询 runtime system 的信息。这些方法允许对象执行内省，比如说 class 方法，要求对象识别它的类; isKindOfClass:和isMemberOfClass:，它测试对象在继承层次结构中的位置; responsesToSelector:，它指示对象是否可以接受特定消息; conformsToProtocol:，它指示对象是否实现特定协议中定义的方法; 和methodForSelector:，它提供了方法实现的地址。这些方法给一个对象有能力内省自己。

Runtime 方法

runtime system 是一个动态共享库，其公共接口由位于目录 /usr/include/ objc 中的头文件中的一组函数和数据结构组成。许多这些函数允许你使用纯 C 来实现编译器在编写 OC 代码时所做的工作。通过 NSObject 类方法导出其他形式的基础功能。这些函数可以开发 runtime system 的其他接口，也可以开发出能增强开发环境的工具，OC 编程中不需要他们。然而，一些 runtime 函数有时候在编写 OC 程序时也很有用。所有这些函数都记录在Objective-C Runtime Reference。

  • 首先，获取选择器指向的程序（方法实现）。因为相同的方法可以被不同的类分别实现，获取的具体程序取决于接收器的类。

消息传递的关键在于编译器为每个类和对象构建了一个结构，每个类结构包括以下两个基本元素：

  • 指向父类的指针
  • 类调度表（dispatch table）。这个表具有将方法选择器与其标识的方法的类特定（class-specific）地址条目关联起来。setOrigin::方法的选择器关联着 setOrigin:: 实现程序的地址，display方法的选择器关联着 display 实现程序的地址等等。

创建新对象时，将为它分配内存，并初始化实例变量。对象的第一个变量是一个指向它类结构的指针。这个指针称为 isa，对象可以通过这个指针访问它的类，并通过类访问继承链上所有的类。

当对象接收到一个消息后，消息传递函数根据对象的 isa 指针找到它的类结构，在类调度表中查找方法选择器，如果没有找到，objc_msgSend 就根据类中指向父类的指针，在父类的调度表中试图找到该选择器。连续查找失败导致 objc_msgSend 爬到类层次结构的顶端 —— NSObject 类。一旦找到与选择器名称相符的方法，就跳至其实现代码。如果最终还是找不到相符的方法，那就执行消息转发（message forwarding）操作。

这是在运行时选择方法实现的方式，即面向对象编程术语中的“方法被动态绑定到消息”。

这么看来，想调用一个方法似乎需要很多步骤。为了加速消息传递过程，runtime system 会把查询结果缓存在“快速映射表”里。每个类都有一个独立的缓存，它即可以包含该类里定义的方法，也可以包含继承下来的方法。在搜索类调度表之前，消息传递例行程序首先检查接受对象的类的“快速映射表”（有个理论说的是，如果一个方法被调用，则很有可能会被再次调用）。如果方法选择器在缓存中，消息传递的速度接近函数调用（静态绑定的函数调用操作）速度。一旦程序运行足够长时间，基本所有的消息都能在缓存中找到。程序运行中，缓存大小会随着新消息的传递而自动扩大。

使用隐藏的参数

当 objc_msgSend 找到方法的实现程序，它便调用这个程序并把消息中的所有参数传递给给它，同时还隐式传递两个参数：

  • 接收对象（receving object）
  • 方法选择器（selector for the method）

这样一来，每个方法的实现都知道自己是“谁的”方法（这就是为什么你能在方法里使用 self 的原因）。之所以说是隐式的，是因为他们没有在方法的源代码中明确的声明，它们是在编译的时候插入到方法实现中的。

尽管这些参数没有被显示的声明，源代码仍然能够引用他们（就好像它能够引用接收对象的实例变量一样）。方法中用 self 来引用接收对象，用 _cmd 来引用选择器。在下面的例子中，_cmd 引用 stange 方法的选择器，self 引用接收 stange 消息的对象。

动态加载

OC 程序能够在运行时加载和链接新的类和类别(catogery)。新代码被合并到程序中，并且与开始时加载的类和类别完全相同。

动态加载被用来做很多不同的事情。例如在系统偏好中的那些模块就是动态加载的。

在 Cocoa 环境中，通常使用动态加载来允许自定义应用程序。你的程序在运行时可以加载别人写的一些模块——就好像 Interface Builder 加载自定义调色板以及 OS X 系统偏好程序加载自定义偏好模块。被加载的程序能够扩展原来程序的功能。他们以一种你允许但又不能预期和定义它的行为的方式提供这些模块。你定义框架，他们实现代码。

尽管在 Mach-O 文件（objc_loadModules, 在 objc/objc-load.h 中定义）中有方法执行 OC 模块的动态加载，Cocoa 的 NSBundle 类为动态绑定提供了一个更方便的接口——它是面向对象的并且集成了很多相关服务。参阅 Foundation framework reference 的 NSBundle 类的说明以获取 NSBundle 类及其用法。关于 Mach-O 的信息请参阅《OS X ABI Mach-O File Format Reference》。

消息转发

给对象发送一个它不能处理的消息会产生一个 error，但在报错之前，runtime system 会给接收消息的对象一个机会来处理消息。

转发

注：这一部分我觉得文档讲的不详细，所以引用《Effective Objective-C 2.0》中“理解消息转发机制”章节的内容。

如果想令类能理解某条消息，我们必须以程序代码实现出对应的方法。但是在编译期向类发送了其无法理解的消息并不会报错，因为在运行期可以继续向类中添加方法，所以编译器在编译时还无法确定类中到底会不会有某个方法的实现。当对象接受到无法解读的消息后就会启动“消息转发”（message forwarding）机制，程序员可以经由此过程来告诉对象应该如何处理未知消息。

在调试程序的时候经常会看到报这样的 warning：

上面这段异常信息由 NSObject 的 “doesNotRecognizeSelector:” 方法抛出，此异常表明：消息接收者的类型为 __NSCFNumber，而该接收者无法理解名为 lowercaseString 的选择器（__NSFNumber 是为了实现无缝桥接而实用的内部类）。这里消息转发过程以应用程序崩溃而gao告终，而开发者在编写自己的类时，可在转发过程中设置挂钩，用以执行预定的逻辑，而不使程序崩溃。

消息转发分为两个阶段：

  • 第一阶段先征询接收者所属的类，看它是否能动态的添加方法来处理当前这个“未知的选择器”，这叫“动态方法解析”（dynamic method resolution）
  • 第二阶段就是进行“完整的消息转发机制”（full forwarding mechanism）。

如果 runtime system 已经把第一阶段执行完，就不能再以动态新增方法的手段来响应包含该选择器的消息了。runtime system 会请求接收者以其他手段来处理与消息相关的方法调用：

  • 首先请接收者看看有没有其他对象能处理这条消息，若有就把消息转给那个对象处理，消息转发结束。
  • 如果没有“备援的接收者”（replacement receiver），则启动完整的消息转发机制，runtime system 会把与消息有关的全部细节封装到 NSInvocation 对象中，再给接收者最后一次机会，令其设法解决当前还未处理的这条消息。

转发的作用

  1. 上面也提到了，转发可以模拟一些多重继承的效果。然而两者有一个显著地不同：多重继承将不同的功能组合在一个对象中，它侧重于大、多层面的对象。而转发侧重于将不同的能力分配给不同的对象。它将问题分解为较小的对象，以一种对消息发送者透明的方式将这些对象联系起来。

  2. 转发可以开发轻量级的代理对象（Surrogate Object），来代表或“覆盖”一些实质性的对象。代理对象对外代表该对象，接收消息并把消息传递给它。《The Objective-C Programming Language》书中“Remote Messaging”章中讨论的 Proxy 就是一种代理。一个 proxy 可以在一个应用中接收消息，然后准备好参数，再将这个消息转发到远程接收器。proxy 并不会复制远程对象的功能，它只给远程对象一个本地地址，通过这个地址远程对象可以接收消息。

  3. 还有其他类别的代理对象。例如你有一个对象，这个对象有大量数据——它可能创建了一个复杂的图像或者读取了磁盘上文件的内容。创建这个对象很耗时，因此你可能倾向于稍后创建——当需要它的时候或者系统资源闲置的时候再创建。同时你需要这个对象的占位符，以便程序中的其他对象可以正常工作。在这种情况下，你可以先创建一个完整的对象，而是一个轻量级的代理。这个对象能够自己处理一些事情，例如回答关于数据的问题，但是它主要的功能还是作为这个大对象的占位符，以及在合适的时候转发消息。当这个代理对象的 forwardInvocation: 方法接收到一个发给其他对象的消息时，它会在该对象不存在时创建它以确保对象的存在。所有发给这个大对象的消息都经过这个代理，对象和代理是等价的。
